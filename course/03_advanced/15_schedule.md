## 本文

### スケジュール実行とは

スケジュール実行は、Claude Code のタスクを**定期的に自動実行**する機能です。Linux の cron ジョブのように、「毎朝 9 時にテストを実行」「毎週月曜日にレポートを生成」といった定期タスクを設定できます。

タスクはリモートセッション（Anthropic クラウド）で実行されるため、パソコンが起動していなくても定時に自動実行されます。

### スケジュールの設定方法

**方法 1：スラッシュコマンドで設定**

```
/schedule 毎朝9時にテストを全件実行して結果をSlackに報告して
```

このコマンドを実行すると、以下が自動設定されます。
- 実行タイミング（cron 式）
- 実行するタスクの内容
- 通知先

**方法 2：対話的に設定**

```
/schedule
```

引数なしで実行すると、対話形式でスケジュールを設定できます。
- 実行頻度の選択
- タスク内容の入力
- 通知設定

### cron 式の基本

スケジュールの実行タイミングは cron 式で指定します。cron 式は 5 つのフィールドで構成されます。

```
┌─── 分 (0-59)
│ ┌─── 時 (0-23)
│ │ ┌─── 日 (1-31)
│ │ │ ┌─── 月 (1-12)
│ │ │ │ ┌─── 曜日 (0-6, 0=日曜)
│ │ │ │ │
* * * * *
```

**よく使う cron 式の例：**

| cron 式 | 意味 |
|---------|------|
| `0 9 * * *` | 毎日 9:00 |
| `0 9 * * 1-5` | 平日（月〜金）9:00 |
| `0 9 * * 1` | 毎週月曜 9:00 |
| `0 */6 * * *` | 6 時間ごと |
| `0 9 1 * *` | 毎月 1 日 9:00 |
| `30 17 * * 5` | 毎週金曜 17:30 |

### 実践的なスケジュール設定例

#### 例 1：毎朝のテスト実行とレポート

```
/schedule
```

設定内容：
- **実行タイミング**：毎朝 8:00（`0 8 * * *`）
- **タスク**：「テストを全件実行して、失敗したテストがあれば原因を分析して修正 PR を作成して。結果を Slack の #ci-reports チャンネルに投稿して」

#### 例 2：週次の依存関係アップデート

```
/schedule 毎週月曜9時にnpmの依存関係をチェックして、セキュリティアップデートがあればアップデートしてPRを作成して
```

設定内容：
- **実行タイミング**：毎週月曜 9:00（`0 9 * * 1`）
- **タスク**：依存関係の脆弱性チェック → アップデート → テスト実行 → PR 作成

#### 例 3：日次のコード品質レポート

```
/schedule 毎日18時にコードベースの品質分析を行って、問題点をNotionのレポートページに追記して
```

#### 例 4：月次のドキュメント更新確認

```
/schedule 毎月1日の10時に、READMEとAPIドキュメントがコードと一致しているか確認して、不一致があれば修正PRを作成して
```

### スケジュールの管理

**スケジュール一覧の確認：**
```
/schedule list
```

設定されているすべてのスケジュールと、その次回実行時刻が表示されます。

**スケジュールの手動実行：**

設定したスケジュールをテスト目的で即座に実行できます。
```
/schedule run <スケジュールID>
```

**スケジュールの更新：**
```
/schedule update <スケジュールID>
```

**スケジュールの削除：**
```
/schedule delete <スケジュールID>
```

### スケジュール実行の仕組み

スケジュール実行は内部的には以下のように動作します。

```
1. 指定された時刻になる
   ↓
2. Anthropic クラウドでリモートセッションが起動
   ↓
3. リポジトリがクローンされる
   ↓
4. 指定されたタスクが実行される
   ↓
5. 結果が保存・通知される
```

リモートセッションとして実行されるため、ローカル環境の設定（ローカルの環境変数、ローカルのデータベースなど）には依存しません。リポジトリに含まれる設定ファイルとシークレット管理の仕組みを使う必要があります。

### コスト管理の注意点

スケジュール実行は自動で定期的にトークンを消費するため、コスト管理が特に重要です。

**コスト最適化のポイント：**

1. **タスクの範囲を絞る**：「コードベース全体をチェック」ではなく「変更があったファイルだけチェック」にする
2. **実行頻度を適切に設定する**：必要以上に高頻度にしない
3. **条件付き実行を検討する**：「変更がある場合のみ実行」のような条件を設定する
4. **`/cost` で定期的に確認する**：予想以上にコストがかかっていないか確認する

### トラブルシューティング

| 問題 | 原因 | 対策 |
|------|------|------|
| スケジュールが実行されない | 認証の期限切れ | API キーの有効期限を確認 |
| タスクが失敗する | リポジトリの権限不足 | リモートセッションからリポジトリにアクセスできるか確認 |
| 結果が通知されない | 通知設定の不備 | MCP コネクタ（Slack 等）の接続を確認 |
| 予想以上にコストがかかる | タスクの範囲が広すぎる | タスクの内容を絞り、完了条件を明確にする |

## やってみよう（ハンズオン）

### ステップ 1：スケジュール一覧を確認する

```
/schedule list
```

現在設定されているスケジュールを確認します（初めての場合は空です）。

### ステップ 2：テスト用のスケジュールを作成する

簡単なスケジュールを設定してみましょう。

```
/schedule 5分後にプロジェクトのファイル数を数えて結果を記録して
```

### ステップ 3：スケジュールの手動実行でテストする

作成したスケジュールを手動で実行して、正しく動作するか確認します。

```
/schedule run <表示されたスケジュールID>
```

### ステップ 4：不要なスケジュールを削除する

テスト用のスケジュールを削除して、クリーンな状態に戻します。

```
/schedule delete <スケジュールID>
```

## まとめ
- スケジュール実行は定期タスクを自動化する機能で、cron ジョブのように使える
- `/schedule` コマンドで設定・管理する
- リモートセッション（クラウド）で実行されるため、パソコンの起動は不要
- テスト実行、依存関係更新、コード品質チェック、ドキュメント更新など幅広く活用できる
- コストが自動で発生するため、タスクの範囲と実行頻度の管理が重要
