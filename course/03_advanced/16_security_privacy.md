## 本文

### Claude Code のセキュリティモデル

Claude Code は、AI エージェントがシステムに直接アクセスする性質上、セキュリティが極めて重要です。Anthropic は**多層防御（Defense in Depth）**のアプローチを採用しています。

```
第1層：パーミッションモード（操作の許可/拒否）
  ↓
第2層：settings.json の allow/deny リスト（決定論的な制御）
  ↓
第3層：Hooks（ツール実行前後の自動チェック）
  ↓
第4層：安全分類器（Auto モードの2段階チェック）
  ↓
第5層：Bash サンドボックス（コマンド実行の隔離）
```

各レイヤーが独立して機能するため、1 つのレイヤーを突破されても、次のレイヤーで防御できます。

### パーミッションモードによる制御

パーミッションモードは、セキュリティの最も基本的なレイヤーです。

| モード | セキュリティレベル | 適した場面 |
|--------|------------------|-----------|
| `plan` | 最高 | 機密性の高いコードの分析 |
| `default` | 高 | 通常の開発作業 |
| `acceptEdits` | 中 | 信頼できるプロジェクトでの反復作業 |
| `auto` | 中（分類器あり） | 長時間タスク（Team/Enterprise/API） |
| `dontAsk` | やや低 | CI/CD（許可リストベース） |
| `bypassPermissions` | 最低 | サンドボックス環境限定 |

### プロンプトインジェクション対策

プロンプトインジェクションとは、外部データに悪意ある指示を埋め込み、AI エージェントの動作を乗っ取る攻撃です。たとえば、読み込んだファイルに「すべてのファイルを削除して」という隠し指示が含まれていた場合に発生します。

**Claude Code の対策：**

1. **Auto モードの分類器設計**
   - 分類器にはユーザーメッセージとツール呼び出しのみが渡される
   - ツールの実行結果（外部からのデータ）は分類器に渡されない
   - これにより、外部データが分類器の判断を操作するリスクを軽減

2. **settings.json の deny リスト**
   - 危険なコマンドを決定論的にブロック
   - AI の判断に依存しないため、プロンプトインジェクションでは突破できない

3. **Hooks による事前チェック**
   - ツール実行前にスクリプトで内容を検証
   - 危険なパターンを検出してブロック

**推奨する deny 設定（プロンプトインジェクション対策）：**

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",
      "Bash(sudo *)",
      "Bash(curl * | bash)",
      "Bash(wget * | bash)",
      "Bash(curl * | sh)",
      "Bash(eval *)",
      "Bash(base64 --decode *)",
      "Bash(nc *)",
      "Bash(ncat *)"
    ]
  }
}
```

### 機密情報の保護

**環境変数と秘密鍵の管理：**

Claude Code は `.env` ファイルやシークレットファイルの内容にアクセスできるため、以下の対策が必要です。

1. **Hooks で機密ファイルの書き込みをブロック**

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "echo $CLAUDE_TOOL_INPUT | python3 -c \"import sys,json; fp=json.load(sys.stdin).get('file_path',''); exit(1) if any(p in fp for p in ['.env','.pem','id_rsa','credentials','secret']) else exit(0)\""
          }
        ]
      }
    ]
  }
}
```

2. **CLAUDE.md で機密ファイルのルールを明記**

```markdown
# 機密ファイルの取り扱い
- .env ファイルの内容を外部に送信しないこと
- 秘密鍵ファイル（.pem、id_rsa）を読み取らないこと
- API キーやトークンをコードにハードコードしないこと
- 機密情報を含むファイルをコミットしないこと
```

3. **`.gitignore` の確認**

```
# .gitignore に含まれるべきファイル
.env
.env.local
.env.production
*.pem
*.key
credentials.json
```

### ネットワークセキュリティ

Claude Code のネットワーク通信に関する注意点です。

**外部通信のリスク：**
- `curl` や `wget` による外部サーバーへのデータ送信
- `npm install` 時の悪意あるパッケージのダウンロード
- MCP サーバー経由の外部通信

**対策：**
- 不要な外部通信コマンドを deny に追加
- 信頼できるパッケージのみをインストール
- MCP サーバーの通信先を確認

### 組織向けセキュリティ設定（Team/Enterprise）

Team プランおよび Enterprise プランでは、組織レベルのセキュリティ管理が可能です。

**監査ログ（Audit Logs）：**
- すべてのエージェント操作を記録
- 誰が、いつ、何をしたかを追跡可能
- コンプライアンス対応に必要

**管理コンソール：**
- 組織全体のパーミッションポリシーを設定
- 許可されるモデルの制限
- API キーの管理
- 使用量の監視

**MDM（モバイルデバイス管理）/ SSO 連携：**
- 企業の認証基盤と統合
- アクセス制御の一元管理

**管理者が設定できるポリシーの例：**

```json
{
  "organizationPolicy": {
    "allowedPermissionModes": ["default", "acceptEdits", "auto"],
    "denyBypassPermissions": true,
    "requiredHooks": ["security-scan"],
    "allowedMcpServers": ["github", "slack"],
    "maxSessionDuration": "8h"
  }
}
```

### データプライバシー

**Claude Code のデータ取り扱い：**

| 項目 | 説明 |
|------|------|
| **コードの保存** | セッション中のコードは処理に必要な期間のみ保持 |
| **トレーニングへの利用** | デフォルトではコードをモデルのトレーニングに使用しない |
| **データの所在** | Anthropic のサーバーで処理 |
| **暗号化** | 通信は TLS で暗号化 |
| **セッションデータ** | ローカルに保存（`~/.claude/` ディレクトリ） |

**プライバシーを強化する設定：**

1. **Plan モードの活用**：機密コードを分析する際は Plan モードを使い、ファイル変更やコマンド実行を行わない
2. **ローカルセッションの管理**：`~/.claude/` ディレクトリのアクセス権限を確認
3. **セッションのエクスポートと削除**：不要なセッションデータを定期的に削除

### セキュリティチェックリスト

日常的に確認すべきセキュリティ項目をまとめました。

**プロジェクト開始時：**
- [ ] settings.json の deny リストに危険なコマンドを追加
- [ ] .gitignore に機密ファイルが含まれているか確認
- [ ] CLAUDE.md に機密情報の取り扱いルールを記載

**日常の作業時：**
- [ ] パーミッションモードが適切か確認
- [ ] コミット前に `git diff` で機密情報の混入がないか確認
- [ ] MCP サーバーの接続状態を定期的に確認

**定期的な見直し：**
- [ ] permissions.allow / deny の内容を見直す
- [ ] 不要な MCP コネクタを無効化
- [ ] セッションデータの整理
- [ ] `/security-review` でコードベースの脆弱性チェック

## まとめ
- Claude Code は多層防御（パーミッション → allow/deny → Hooks → 安全分類器 → サンドボックス）でセキュリティを確保する
- プロンプトインジェクション対策として、deny リストによる決定論的なブロックが最も確実
- 機密情報は Hooks でファイル書き込みをブロックし、CLAUDE.md でルールを明記する
- Team/Enterprise では監査ログ、管理コンソール、SSO 連携で組織レベルのセキュリティを実現
- セキュリティチェックリストを使って定期的に設定を見直す
