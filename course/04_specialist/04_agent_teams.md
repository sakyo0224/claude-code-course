## 本文

### なぜ複数エージェントが必要なのか

単一の Claude Code セッションでも多くのタスクを処理できますが、以下のような場面では限界があります。

1. **コンテキストの肥大化**: 1 つのセッションに設計・実装・テスト・レビューすべてを詰め込むと、コンテキストウィンドウが圧迫される
2. **並列作業の必要性**: フロントエンドとバックエンドを同時に開発したい場合、1 セッションでは逐次処理になる
3. **専門性の分離**: セキュリティレビューとコード実装では、必要な指示やコンテキストが大きく異なる

Agent Teams は、これらの課題を「複数のエージェントが協調して動作する」ことで解決します。

### Agent Teams の 2 つのアプローチ

Claude Code で複数エージェントを活用するには、主に 2 つの方法があります。

**アプローチ 1: サブエージェント（`/agents`）**

サブエージェントは、メインのセッションから呼び出される専門的なエージェントです。`/agents` コマンドで管理します。

```
/agents
```

サブエージェントは以下のように定義します。

```json
// .claude/agents/reviewer.json
{
  "name": "reviewer",
  "description": "コードレビュー専門のエージェント",
  "instructions": "変更されたファイルのコードレビューを行い、問題点と改善提案を報告してください。セキュリティ、パフォーマンス、可読性の観点でチェックしてください。",
  "tools": ["Read", "Glob", "Grep"],
  "permissionMode": "plan"
}
```

```json
// .claude/agents/tester.json
{
  "name": "tester",
  "description": "テスト作成と実行の専門エージェント",
  "instructions": "指定されたコードに対してユニットテストを作成し、実行してください。カバレッジ80%以上を目標にしてください。",
  "tools": ["Read", "Write", "Edit", "Bash"],
  "permissionMode": "acceptEdits"
}
```

メインセッションから以下のように呼び出します。

```
@reviewer src/api/users.ts の変更をレビューして
@tester src/api/users.ts のテストを書いて実行して
```

**アプローチ 2: Teammate モード（`--teammate-mode`）**

`--teammate-mode` は、複数の Claude Code セッションが互いに通信しながら協調動作するモードです。Git worktree を使ってファイルシステムの競合を避けます。

```bash
# ターミナル 1: フロントエンド担当
claude --teammate-mode --name "frontend"

# ターミナル 2: バックエンド担当
claude --teammate-mode --name "backend"

# ターミナル 3: テスト担当
claude --teammate-mode --name "tester"
```

各エージェントは独立した Git worktree で作業するため、ファイルの競合が発生しません。完了後にブランチをマージすることで、全体の変更を統合します。

### 役割分担パターン

**パターン 1: 設計 → 実装 → レビュー（直列型）**

```
設計エージェント → 実装エージェント → レビューエージェント
  （Plan mode）     （Auto mode）      （Plan mode）
```

1. 設計エージェントが方針と設計書を作成
2. 実装エージェントが設計書に基づいてコードを実装
3. レビューエージェントが実装をチェックし、問題があればフィードバック

**パターン 2: フロントエンド + バックエンド（並列型）**

```
         ┌── フロントエンド担当 ──┐
タスク定義 ┤                       ├── 統合テスト
         └── バックエンド担当 ────┘
```

API の仕様（OpenAPI 定義など）を事前に共有し、フロントエンドとバックエンドを並列に開発します。

**パターン 3: 実装 + 監視（常駐型）**

```
実装エージェント ←→ 品質監視エージェント（/loop で常駐）
```

`/loop` コマンドで品質監視エージェントを定期実行し、実装エージェントが生成したコードを継続的にチェックします。

### 協調動作の設計指針

1. **共有ファイルで通信する**: エージェント間の情報共有は、ファイルシステム上の共有ドキュメント（設計書、API 仕様書、TODO リストなど）を介して行う
2. **コンテキストを最小化する**: 各エージェントには、その役割に必要な最小限のコンテキストだけを与える
3. **権限を最小化する**: レビュー担当エージェントには `plan` モード（読み取り専用）、実装担当には `acceptEdits` モードを割り当てる
4. **Git ブランチで作業を分離する**: 並列作業の場合、各エージェントは別々のブランチで作業し、最後にマージする

### 実践的な構成例

中規模の Web アプリケーション開発での構成例を示します。

```
project/
├── .claude/
│   ├── agents/
│   │   ├── architect.json    # 設計担当
│   │   ├── implementer.json  # 実装担当
│   │   ├── reviewer.json     # レビュー担当
│   │   └── tester.json       # テスト担当
│   └── settings.json
├── CLAUDE.md                  # 全エージェント共通のルール
└── docs/
    ├── architecture.md        # 設計書（architect が作成、他が参照）
    ├── api-spec.yaml          # API 仕様（共有）
    └── review-checklist.md    # レビューチェックリスト
```

## やってみよう（ハンズオン）

### ステップ 1: サブエージェントを定義する

`.claude/agents/` ディレクトリに、`reviewer.json` を作成してください。内容は本文中の例を参考にしてください。

### ステップ 2: サブエージェントを呼び出す

メインセッションから `@reviewer` を使って、プロジェクト内の任意のファイルのレビューを依頼してみましょう。

### ステップ 3: 並列セッションを試す

2 つのターミナルを開き、それぞれ `--teammate-mode` で起動して、異なるタスクを並列に実行してみてください。

---

## まとめ
- Agent Teams は、コンテキストの肥大化・並列作業・専門性の分離という課題を解決する
- サブエージェント（`/agents`）は 1 セッション内で専門家を呼び出す方法
- Teammate モード（`--teammate-mode`）は複数セッションが Git worktree で協調する方法
- 各エージェントには最小限のコンテキストと最小限の権限を与えるのが設計の基本
