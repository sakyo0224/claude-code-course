## 本文

### なぜ監査ログが重要なのか

AI エージェントがコードを自動で編集し、コマンドを実行し、外部サービスと通信する環境では、「誰が」「いつ」「何を」指示し、「エージェントが何を実行したか」を記録することが不可欠です。

監査ログは以下の目的で使用されます。

1. **セキュリティインシデントの調査**: 不正な操作や情報漏洩が発生した場合の原因究明
2. **コンプライアンス証拠**: 規制当局への報告や内部監査への対応
3. **利用状況の分析**: 組織全体の利用パターンを把握し、最適化に活用
4. **問題の早期発見**: 異常な操作パターンを検知し、被害を未然に防ぐ

### Claude Code の監査ログ機能

**重要な前提**: 監査ログは **Team プランおよび Enterprise プラン** でのみ利用可能です。Pro/Max プランでは監査ログ機能は提供されていません。また、Cowork（リサーチプレビュー）は監査ログに対応していません。

Claude Code の監査ログには以下の情報が記録されます。

| 記録項目 | 説明 |
|---------|------|
| ユーザー ID | 操作を行ったメンバーの識別子 |
| タイムスタンプ | 操作の日時（UTC） |
| セッション ID | Claude Code セッションの識別子 |
| ツール呼び出し | 使用されたツール名（Bash、Edit、Read など） |
| ツール入力 | ツールに渡されたパラメータ |
| ツール出力 | ツールの実行結果（設定により省略可能） |
| パーミッション判定 | 操作が許可/拒否/確認のいずれだったか |
| モデル | 使用されたモデル名 |
| トークン消費 | 入力/出力トークン数 |

### 監査ログの取得方法

**方法 1: 管理コンソールからの確認**

Team/Enterprise プランの管理コンソールにログインし、監査ログセクションから閲覧・検索・エクスポートが可能です。

**方法 2: Compliance API による取得**

プログラムで監査ログを取得する場合は、Compliance API を使用します。

```bash
# 監査ログの取得例（API）
curl -H "Authorization: Bearer $ADMIN_API_KEY" \
  "https://api.anthropic.com/v1/organizations/{org_id}/audit-logs?start_date=2026-04-01&end_date=2026-04-06"
```

**方法 3: Data Exports（データエクスポート）**

定期的にログデータを一括エクスポートし、社内の SIEM（セキュリティ情報イベント管理）ツールに取り込むことも可能です。

### コンプライアンス対応のチェックリスト

主要な規制・フレームワークごとの対応ポイントを整理します。

**SOC 2 対応**

```markdown
- [ ] アクセス制御: パーミッションモードで操作権限を制限
- [ ] 監査ログ: 全操作のログ記録を有効化
- [ ] 変更管理: Git によるコード変更の追跡
- [ ] インシデント対応: 異常検知と通知の仕組みを構築
- [ ] 暗号化: 通信の暗号化（TLS）、保存データの暗号化
```

**GDPR 対応**

```markdown
- [ ] データ処理契約（DPA）の締結
- [ ] 個人データの取り扱い範囲の明確化
- [ ] データ保持期間の設定
- [ ] データ主体の権利（アクセス権、削除権）への対応
- [ ] プライバシー影響評価（DPIA）の実施
```

**社内セキュリティポリシー対応**

```markdown
- [ ] 機密情報の分類基準との整合性確認
- [ ] 認証・認可の要件（SSO、MFA）の充足
- [ ] ネットワークセキュリティの要件確認
- [ ] 定期的なセキュリティレビューのスケジュール設定
```

### ログ分析と異常検知

監査ログを活用して、以下のような異常パターンを検知できます。

**パターン 1: 大量のファイルアクセス**

短時間に多数のファイルを読み取る操作は、情報収集の試みを示す可能性があります。

```
検知条件: 1 セッションで 100 ファイル以上の Read が 10 分以内に発生
対応: アラート通知 → セッションの内容を確認
```

**パターン 2: 通常と異なる時間帯の利用**

深夜や休日の利用は、不正アクセスの兆候である可能性があります。

```
検知条件: 営業時間外（22:00〜6:00）のセッション開始
対応: アラート通知 → ユーザーへの確認
```

**パターン 3: 権限拒否の頻発**

permissions.deny に設定された操作が繰り返し試行される場合、設定を回避しようとしている可能性があります。

```
検知条件: 同一ユーザーで deny 操作が 1 時間に 5 回以上
対応: アラート通知 → ポリシーの見直しまたはユーザーへの教育
```

### Hooks を使ったリアルタイム監視

Hooks を組み合わせることで、リアルタイムの監視と通知を実現できます。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "command": "scripts/audit-bash-command.sh"
      }
    ],
    "Notification": [
      {
        "matcher": "",
        "command": "scripts/send-slack-alert.sh"
      }
    ]
  }
}
```

```bash
#!/bin/bash
# scripts/audit-bash-command.sh
COMMAND="$1"
TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
USER=$(whoami)

# ローカル監査ログに記録
echo "$TIMESTAMP | $USER | Bash | $COMMAND" >> /var/log/claude-code-audit.log

# 危険なコマンドの場合は Slack に通知
if echo "$COMMAND" | grep -qE "(rm -rf|sudo|curl.*\|.*bash)"; then
  curl -X POST "$SLACK_WEBHOOK_URL" \
    -d "{\"text\": \"⚠️ 危険なコマンド検知: $USER が '$COMMAND' を実行しました\"}"
fi
```

## やってみよう（ハンズオン）

### ステップ 1: 監査ログの確認

Team/Enterprise プランをお使いの場合は、管理コンソールにログインし、過去 1 週間の監査ログを確認してください。個人プランの場合は、Hook を使ったローカル監査ログの仕組みを構築しましょう。

### ステップ 2: コンプライアンスチェックリストの作成

あなたの組織に適用される規制やポリシーに合わせて、Claude Code 利用のコンプライアンスチェックリストを作成してください。

### ステップ 3: 異常検知ルールの設計

あなたの組織にとって「異常」とみなすべき操作パターンを 3 つ以上定義し、検知条件と対応方法をドキュメント化してください。

---

## まとめ
- 監査ログは Team/Enterprise プラン限定の機能で、全操作の記録・追跡が可能
- SOC 2・GDPR などのコンプライアンス対応には、監査ログの有効化が前提条件
- ログ分析により、大量ファイルアクセス・異常時間帯・権限拒否の頻発などの異常を検知できる
- Hooks を活用してリアルタイムの監視・通知を構築できる
