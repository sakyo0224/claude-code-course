## 本文

### プランの比較

Claude Code は個人向けの Pro/Max プランに加え、組織向けの Team プランと Enterprise プランを提供しています。

| 項目 | Pro | Max | Team | Enterprise |
|------|-----|-----|------|-----------|
| 対象 | 個人 | 個人（ヘビーユーザー） | チーム（小〜中規模） | 大規模組織 |
| Auto モード | 不可 | 不可 | 利用可能 | 利用可能 |
| 監査ログ | なし | なし | あり | あり（詳細） |
| SSO/SAML | なし | なし | あり | あり |
| 管理コンソール | なし | なし | あり | あり（MDM対応） |
| SLA | なし | なし | あり | カスタム |
| データ保持ポリシー | 標準 | 標準 | カスタマイズ可 | 完全カスタマイズ |

### Team プラン導入の判断基準

以下のいずれかに該当する場合、Team プランの導入を検討してください。

1. **3 人以上の開発者が Claude Code を日常的に使用している**: 個別の Pro 契約よりもコスト効率が良い場合がある
2. **Auto モードを使いたい**: 長時間の自走タスクで許可疲れを解消したい
3. **利用状況を把握したい**: 誰がどのくらいのトークンを消費しているかを可視化する必要がある
4. **セキュリティポリシーの適用が必要**: 全メンバーに統一された権限設定を適用したい

### Enterprise プラン導入の判断基準

以下のいずれかに該当する場合、Enterprise プランが必要です。

1. **SSO/SAML が必須**: 社内の ID プロバイダーとの統合が必要
2. **MDM（モバイルデバイス管理）が必要**: デスクトップアプリの配布と設定を一元管理したい
3. **詳細な監査ログが必要**: コンプライアンス要件で全操作のログ記録が求められる
4. **カスタム SLA が必要**: 稼働率保証やサポート対応時間の合意が必要
5. **データ処理契約（DPA）が必要**: GDPR やその他の規制で DPA の締結が必要

### 組織展開の設計

**ステップ 1: ガバナンスポリシーの策定**

まず、組織として以下を決定します。

```markdown
## Claude Code 利用ポリシー（例）

### 許可する操作
- プロジェクトディレクトリ内のファイル読み書き
- テスト実行
- Git 操作（フィーチャーブランチのみ）

### 禁止する操作
- 本番環境への直接デプロイ
- main/master ブランチへの直接プッシュ
- 機密データ（個人情報、認証情報）を含むファイルへのアクセス

### パーミッションモードの基準
- 新規メンバー: default（確認モード）
- 経験者: acceptEdits（編集自動承認）
- シニアメンバー: auto（Auto モード）
```

**ステップ 2: 共有設定の配布**

プロジェクトの `.claude/settings.json` に組織共通の設定を定義し、Git リポジトリで管理します。

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Edit",
      "Write",
      "Glob",
      "Grep",
      "Bash(git *)",
      "Bash(npm test)",
      "Bash(npm run lint)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(sudo *)",
      "Bash(git push origin main)",
      "Bash(git push origin master)"
    ]
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "command": "scripts/org-lint-check.sh"
      }
    ]
  }
}
```

**ステップ 3: Auto モードの有効化**

Team/Enterprise プランでは、管理コンソールから Auto モードを有効化できます。有効化の際は以下を確認してください。

- 利用できるモデル（Claude Sonnet 4.6 または Opus 4.6）が設定されている
- プロバイダが Anthropic API であること
- 管理者が事前に Auto モードの利用を承認している

Auto モードの安全分類器は以下のように動作します。

1. **Stage 1**: 高速な Yes/No フィルターで明らかに安全/危険な操作を分類
2. **Stage 2**: フラグされた操作に対して、思考連鎖による詳細判断を実施

### オンボーディングの設計

新しいメンバーが Claude Code を使い始める際のオンボーディングフローを用意しましょう。

```
1. アカウント発行（管理者が招待）
2. デスクトップアプリのインストール
3. 初期パーミッションモード: default（確認モード）
4. CLAUDE.md の読み合わせ
5. 基本操作のハンズオン（ファイル編集、テスト実行）
6. 1 週間後: acceptEdits モードへの移行判断
7. 1 ヶ月後: Auto モードへの移行判断（該当プランの場合）
```

### コスト管理

Team/Enterprise プランでは、管理コンソールからメンバーごとのトークン消費を確認できます。以下の指標を定期的にモニタリングしましょう。

- **メンバーごとの日次/週次消費量**: 平均を大きく超えているメンバーがいないか
- **プロジェクトごとの消費量**: 特定のプロジェクトで消費が集中していないか
- **消費のトレンド**: 月を追うごとに消費が増加していないか

## やってみよう（ハンズオン）

### ステップ 1: 利用ポリシーのドラフトを作成する

あなたの組織に合わせた「Claude Code 利用ポリシー」のドラフトを作成してください。本文中の例を参考に、許可する操作・禁止する操作・パーミッションモードの基準を記述します。

### ステップ 2: 共有 settings.json を設計する

ポリシーに基づいて `.claude/settings.json` を作成し、`permissions.allow` と `permissions.deny` を設定してください。

### ステップ 3: オンボーディング資料を作成する

新しいメンバー向けに、Claude Code のセットアップから基本操作までの手順書を作成してください。

---

## まとめ
- Team プランは Auto モード・監査ログ・管理コンソールが必要なチーム向け
- Enterprise プランは SSO/SAML・MDM・詳細監査ログ・カスタム SLA が必要な大規模組織向け
- 組織展開の基本は「ガバナンスポリシー策定 → 共有設定の配布 → Auto モードの段階的有効化」
- オンボーディングでは、パーミッションモードを段階的に移行させるのが安全
