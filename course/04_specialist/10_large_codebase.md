## 本文

### 大規模コードベースの課題

コードベースが数十万行を超えると、以下の課題が顕著になります。

1. **全体像の把握が困難**: Claude のコンテキストウィンドウに全ファイルを収めることができない
2. **検索結果の氾濫**: `Grep` や `Glob` の結果が大量に返され、関連する部分を絞り込むのが難しい
3. **依存関係の複雑さ**: ある変更が他のモジュールに与える影響を正確に把握しにくい
4. **ビルド時間**: 変更のたびに全体をビルドすると時間がかかる

### 戦略 1: CLAUDE.md によるナビゲーションマップ

大規模コードベースでは、CLAUDE.md を「地図」として機能させることが極めて重要です。

```markdown
# CLAUDE.md（プロジェクトルート）

## プロジェクト構造
このプロジェクトは以下のモジュールで構成されています。

- `packages/core/` — ビジネスロジックの中核。依存関係なし
- `packages/api/` — REST API サーバー。core に依存
- `packages/web/` — フロントエンド（React）。api を呼び出す
- `packages/cli/` — CLI ツール。core に依存
- `packages/shared/` — 共有型定義とユーティリティ

## 主要なエントリポイント
- API サーバー: `packages/api/src/index.ts`
- Web アプリ: `packages/web/src/App.tsx`
- CLI: `packages/cli/src/main.ts`

## よく変更するファイル
- ルーティング: `packages/api/src/routes/`
- データモデル: `packages/core/src/models/`
- UI コンポーネント: `packages/web/src/components/`

## テスト実行
- 全体: `npm test`
- パッケージ単位: `npm test -w packages/core`
- 単一ファイル: `npx vitest run src/path/to/file.test.ts`
```

この地図があることで、Claude は「どのディレクトリに何があるか」を初手で理解し、的確にファイルを探索できます。

### 戦略 2: スコープの明示的な制限

大規模コードベースでは、「全体を見て」という指示を避け、常にスコープを明示します。

```
# 悪い例
「このプロジェクトのバグを直して」

# 良い例
「packages/api/src/routes/users.ts の認証ミドルウェアで、
 トークンの有効期限チェックが正しく動作しない問題を修正して」
```

スコープを絞る際のポイントは以下のとおりです。

1. **ディレクトリを指定する**: 「`packages/api/` 内で」
2. **ファイルを指定する**: 「`src/routes/users.ts` を」
3. **行番号を指定する**: 「100〜150 行目あたりの」
4. **機能を指定する**: 「ユーザー認証の」

### 戦略 3: `/add-dir` による作業範囲の制御

`/add-dir` コマンドで、Claude Code の作業ディレクトリを明示的に追加できます。大規模コードベースでは、必要なディレクトリだけを追加することで、検索の効率を高められます。

```
/add-dir packages/core
/add-dir packages/api
```

これにより、Claude は指定されたディレクトリ内でのみファイル検索を行います。

### 戦略 4: Git worktree による並列作業

大規模プロジェクトでは、異なるモジュールの変更を並列に進めたいことがよくあります。Git worktree を使えば、同じリポジトリの複数のブランチを同時に操作できます。

```bash
# worktree を作成
git worktree add ../project-api-fix feature/api-fix
git worktree add ../project-web-update feature/web-update

# 別々のターミナルで Claude Code を起動
# ターミナル 1
cd ../project-api-fix && claude

# ターミナル 2
cd ../project-web-update && claude
```

それぞれのセッションは独立したファイルシステムで動作するため、競合が発生しません。

### 戦略 5: サブエージェントによる分割統治

大規模な変更を 1 つのセッションで行うのではなく、サブエージェントに役割を分担させます。

```
メインエージェント（指揮者）
  ├── @api-agent: API 層の変更を担当
  ├── @web-agent: フロントエンドの変更を担当
  ├── @test-agent: テストの作成と実行を担当
  └── @reviewer: 全体の整合性をチェック
```

各サブエージェントには、担当するディレクトリとその CLAUDE.md だけがコンテキストとして渡されるため、コンテキストウィンドウの効率が良くなります。

### 戦略 6: `/plan` と `/ultraplan` の活用

大規模な変更を実行する前に、まず計画を立てましょう。

```
/plan packages/core のデータモデルを v2 に移行する計画を立てて
```

特に広範囲に影響する変更の場合は `/ultraplan` を使います。

```
/ultraplan データベーススキーマの変更とそれに伴う全モジュールの修正計画
```

`/ultraplan` はブラウザ上で計画をレビューできるため、複雑な変更の全体像を把握しやすくなります。

### 戦略 7: リモートセッションの活用

大規模コードベースでの長時間タスク（リファクタリング、マイグレーションなど）は、リモートセッションで実行するのが効果的です。

```
# リモートセッションを開始
# パソコンを閉じても作業が継続される
```

リモートセッションでは複数リポジトリを同時に扱えるため、モノレポのクロスモジュール変更にも適しています。

### モノレポでの運用パターン

モノレポ（1 つのリポジトリに複数のパッケージが含まれる構成）では、以下のパターンが有効です。

```
monorepo/
├── CLAUDE.md                    # 全体の地図
├── .claude/
│   ├── settings.json            # 共通設定
│   └── agents/
│       ├── core-dev.json        # core パッケージ担当
│       └── web-dev.json         # web パッケージ担当
├── packages/
│   ├── core/
│   │   └── CLAUDE.md            # core 固有の情報
│   ├── api/
│   │   └── CLAUDE.md            # api 固有の情報
│   └── web/
│       └── CLAUDE.md            # web 固有の情報
```

各パッケージの CLAUDE.md には、そのパッケージの責務・依存関係・テスト方法だけを記載します。

## やってみよう（ハンズオン）

### ステップ 1: ナビゲーションマップを作成する

あなたのプロジェクトの CLAUDE.md に、プロジェクト構造・主要エントリポイント・よく変更するファイルを記述してください。

### ステップ 2: スコープを絞った指示を実践する

普段「ざっくり」と指示していたタスクを、ディレクトリ・ファイル・行番号を明示した指示に書き換えて実行し、応答の精度と速度の違いを確認してください。

### ステップ 3: Git worktree で並列作業を試す

2 つの worktree を作成し、それぞれで Claude Code を起動して、異なるモジュールの変更を並列に進めてみてください。

---

## まとめ
- 大規模コードベースでは CLAUDE.md を「ナビゲーションマップ」として機能させる
- 指示には常にスコープ（ディレクトリ、ファイル、行番号）を明示する
- Git worktree とサブエージェントで並列作業を実現する
- `/plan` や `/ultraplan` で大規模変更の計画を事前に立てる
- リモートセッションで長時間タスクを安定して実行する
