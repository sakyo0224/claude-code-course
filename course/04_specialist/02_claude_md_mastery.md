## 本文

### CLAUDE.md の読み込み階層

Claude Code は起動時に複数の CLAUDE.md を自動で読み込みます。その優先順位は以下のとおりです。

```
1. ~/.claude/CLAUDE.md          （ユーザーグローバル — 全プロジェクト共通）
2. リポジトリルート/CLAUDE.md    （プロジェクト全体）
3. リポジトリルート/.claude/CLAUDE.md （プロジェクト全体、隠しディレクトリ版）
4. カレントディレクトリ/CLAUDE.md （サブディレクトリ固有）
5. 親ディレクトリの CLAUDE.md    （ディレクトリ階層を遡って読み込み）
```

すべてのファイルが結合されてコンテキストに含まれるため、重複する記述は避けるべきです。階層が深いファイルほど「そのディレクトリ固有の情報」に絞ることで、トークン効率を高められます。

### 設計原則: CLAUDE.md vs settings.json

まず、CLAUDE.md と settings.json の使い分けを明確にしましょう。

| 特性 | CLAUDE.md | settings.json |
|------|-----------|---------------|
| 性質 | 自然言語の指示（ソフトルール） | JSON による設定（ハードルール） |
| 決定論性 | 確率的（モデルが解釈） | 決定論的（システムが強制） |
| 用途 | コーディング規約、手順、背景知識 | パーミッション、モデル選択、ツール設定 |
| Git 管理 | リポジトリに含める（共有推奨） | `.claude/settings.json` はリポジトリに含める |

**原則**: 「必ず守らせたいルール」は settings.json の `permissions.deny` で強制し、「参考にしてほしいガイドライン」は CLAUDE.md に書きます。

### 高度な記述パターン

**パターン 1: 条件付き指示**

```markdown
## コーディング規約
- TypeScript を使用する
- 関数には JSDoc コメントを必ず付ける
- テストファイルは `*.test.ts` の命名規則に従う

## テストについて
- テストを書く際は `vitest` を使用する
- テスト実行コマンド: `npm test`
- テストを書いたら必ず実行して通ることを確認する

## 重要な制約
- `main` ブランチに直接コミットしない。必ずフィーチャーブランチを作成する
- データベースマイグレーションを含む変更は、必ず実行計画を先に提示する
```

**パターン 2: ディレクトリ固有の CLAUDE.md**

```
project/
├── CLAUDE.md                    # プロジェクト全体の規約
├── src/
│   ├── CLAUDE.md                # フロントエンド固有のルール
│   └── components/
│       └── CLAUDE.md            # コンポーネント設計のガイドライン
├── server/
│   └── CLAUDE.md                # バックエンド固有のルール
└── scripts/
    └── CLAUDE.md                # スクリプト実行時の注意事項
```

各ディレクトリの CLAUDE.md には、そのディレクトリに関する情報だけを書きます。

```markdown
# server/CLAUDE.md
- このディレクトリは Express.js バックエンドのコードを含む
- データベースは PostgreSQL を使用
- API エンドポイントは RESTful 設計に従う
- 環境変数は `.env` から読み込む（.env ファイルは Git に含めないこと）
```

**パターン 3: 動的コンテキストの参照**

CLAUDE.md からプロジェクト内の他のドキュメントを参照させることで、必要な情報を段階的に開示できます。

```markdown
## アーキテクチャ
- 全体設計は `docs/architecture.md` を参照してください
- API 仕様は `docs/api-spec.yaml` に定義されています

## デプロイ手順
- 本番デプロイの手順は `docs/deploy-guide.md` に記載されています
- デプロイ前に必ず確認してください
```

この書き方であれば、CLAUDE.md 自体は軽量に保ちながら、必要なときにだけ詳細情報を読み込ませることができます。

**パターン 4: エラー回復の指示**

```markdown
## エラー対処
- TypeScript のコンパイルエラーが出た場合、まず `npx tsc --noEmit` で全体の型チェックを行う
- テスト失敗時は、失敗したテストだけを `npm test -- --filter "テスト名"` で再実行する
- lint エラーは `npm run lint:fix` で自動修正を試み、修正できないものは手動で対処する
```

これにより、Claude がエラーに遭遇した際に「何をすべきか」を明確に知ることができ、無駄な試行錯誤（＝トークン消費）を減らせます。

### チーム開発での運用ルール

チームで CLAUDE.md を運用する際は、以下のルールを設けることをおすすめします。

1. **プロジェクトルートの CLAUDE.md はチームで合意した内容のみを記載**: 個人的な好みはユーザーグローバル（`~/.claude/CLAUDE.md`）に書く
2. **PR レビューの対象に含める**: CLAUDE.md の変更もコードと同様にレビューする
3. **定期的に棚卸しする**: 月に 1 回程度、古くなった記述や矛盾する記述がないかをチームで確認する
4. **バージョン管理する**: CLAUDE.md は必ず Git で管理し、変更履歴を追跡できるようにする

### アンチパターン

避けるべき記述パターンを紹介します。

```markdown
# 悪い例 1: 曖昧すぎる指示
- 良いコードを書いてください

# 悪い例 2: 矛盾する指示
- 関数は短くシンプルに保つこと
- すべてのエッジケースを関数内で処理すること

# 悪い例 3: 過度に詳細な背景説明
- このプロジェクトは2023年に始まり、当初はPythonで書かれていましたが
  2024年にTypeScriptに移行し、チームは5人から12人に拡大し…（500字続く）
```

## やってみよう（ハンズオン）

### ステップ 1: 現在の CLAUDE.md を監査する

プロジェクトの CLAUDE.md を開き、以下の観点でチェックしてください。

- [ ] 曖昧な表現がないか
- [ ] 矛盾する指示がないか
- [ ] 不要な背景説明が含まれていないか
- [ ] settings.json で強制すべきルールが混在していないか

### ステップ 2: ディレクトリ階層に分割する

プロジェクトルートの CLAUDE.md から、ディレクトリ固有の情報を切り出して子ディレクトリの CLAUDE.md に移動してください。

### ステップ 3: トークン消費の変化を測定する

分割前と分割後で `/cost` の値を比較し、トークン消費がどのように変化したかを記録してください。

## まとめ
- CLAUDE.md は複数階層で読み込まれるため、重複を避けて階層ごとに情報を分離する
- 「ソフトルール」は CLAUDE.md に、「ハードルール」は settings.json に書く
- 動的コンテキストの参照を使い、CLAUDE.md 自体は軽量に保つ
- チーム運用では PR レビューの対象に含め、定期的に棚卸しする
