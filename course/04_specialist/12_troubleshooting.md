# トラブルシューティング完全ガイド

## この記事で学ぶこと
- Claude Code で発生しやすい問題とその根本原因
- 問題を体系的に診断・解決する手順
- 予防策を設定に組み込む方法

## 前提知識
- Claude Code の基本操作から応用まで一通り経験していること
- settings.json、CLAUDE.md、Hooks の仕組みを理解していること

---

## 本文

### トラブルシューティングの基本フレームワーク

問題が発生したときは、以下の 4 ステップで対処します。

```
1. 症状を正確に把握する（何が起きているか）
2. 原因を特定する（なぜ起きているか）
3. 解決策を実行する（どう直すか）
4. 再発を防止する（どう防ぐか）
```

### カテゴリ 1: 接続・認証の問題

**症状: Claude Code が起動しない、または「認証に失敗しました」と表示される**

| 原因 | 対処法 |
|------|--------|
| API キーの期限切れ | `claude config` で API キーを再設定する |
| ネットワーク接続の問題 | プロキシ設定を確認する。VPN を使用している場合は一時的にオフにして試す |
| プランの期限切れ | claude.ai にログインしてプランの状態を確認する |
| デスクトップアプリのバージョンが古い | 最新版にアップデートする |

**診断コマンド**

```
/doctor
```

`/doctor` コマンドは Claude Code のインストール状態を診断し、問題があれば解決策を提示します。まず最初に実行してください。

### カテゴリ 2: パフォーマンスの問題

**症状: 応答が遅い、タイムアウトする**

```
原因 1: コンテキストウィンドウの肥大化
  → /compact で会話を圧縮する
  → /context で現在のコンテキスト使用率を確認する

原因 2: 大きなファイルの読み込み
  → ファイルの必要な部分だけを読み込むよう指示する
  → 「100〜150行目を読んで」のように範囲指定する

原因 3: 重い Hook スクリプトの実行
  → /hooks で登録されている Hook を確認する
  → Hook スクリプトの実行時間を計測し、最適化する

原因 4: モデルの負荷
  → /effort low に変更して応答速度を優先する
  → /fast on で高速モードを有効にする
```

**症状: トークン消費が異常に多い**

```
確認手順:
1. /cost で現在の消費量を確認
2. /context でコンテキストの内訳を確認
3. 大きなファイルや画像がコンテキストに含まれていないか確認
4. /compact で不要な履歴を圧縮

予防策:
- settings.json に autoCompactThreshold を設定する
- CLAUDE.md を簡潔に保つ
- セッションを適切に分割する
```

### カテゴリ 3: ファイル操作の問題

**症状: ファイルの編集が期待どおりにならない**

| 原因 | 対処法 |
|------|--------|
| 編集箇所の特定ミス | 行番号やコンテキストを具体的に指示する |
| 文字コードの問題 | ファイルの文字コード（UTF-8 など）を明示する |
| 改行コードの混在 | `.gitattributes` で改行コードを統一する |
| ファイルの権限不足 | ファイルのパーミッションを確認する |

**症状: Git の操作でエラーが出る**

```
よくあるケース:

1. 「ブランチが存在しない」
   → git branch -a で全ブランチを確認
   → リモートブランチを fetch してから操作

2. 「マージコンフリクト」
   → Claude Code にコンフリクトの解決を依頼
   → 「コンフリクトを解決して、feature ブランチの変更を優先して」

3. 「push が拒否された」
   → リモートの最新を pull してから push
   → permissions.deny に git push が含まれていないか確認
```

### カテゴリ 4: MCP サーバーの問題

**症状: MCP サーバーに接続できない**

```
診断手順:
1. /mcp で接続状態を確認
2. settings.json の mcpServers 設定を確認
3. MCP サーバーのプロセスが起動しているか確認
4. 環境変数が正しく設定されているか確認

よくある原因:
- command のパスが間違っている（絶対パスを使用すべき）
- 環境変数が未設定（$API_TOKEN など）
- Node.js/Python のバージョンが要件を満たしていない
- ポートが他のプロセスに使用されている
```

**症状: MCP ツールが認識されない**

```
確認手順:
1. /mcp でツール一覧を表示し、期待するツールがあるか確認
2. MCP サーバーの定義でツール名が正しいか確認
3. MCP サーバーを再起動する（Claude Code を再起動）
```

### カテゴリ 5: Hooks の問題

**症状: Hook が実行されない**

```
確認手順:
1. /hooks で登録状況を確認
2. スクリプトに実行権限があるか確認（chmod +x）
3. スクリプトのパスが正しいか確認
4. matcher の文字列がツール名と一致しているか確認

よくある原因:
- 実行権限がない: chmod +x scripts/my-hook.sh
- シェバン行が欠落: #!/bin/bash を先頭に追加
- matcher のスペルミス: "Edit" ではなく "edit" と書いている
```

**症状: Hook が無限ループする**

```
原因: Hook の出力が Claude に新しいツール呼び出しを促し、
      それがまた Hook を発火させるループ

対処:
- Hook スクリプト内で環境変数を使ってループを検知する
- 例: HOOK_RUNNING=1 を設定し、2回目以降は早期 return
```

### カテゴリ 6: Auto モードの問題

**症状: Auto モードが利用できない**

```
前提条件の確認:
- プラン: Team、Enterprise、または API（Pro/Max では不可）
- モデル: Claude Sonnet 4.6 または Opus 4.6
- プロバイダ: Anthropic API のみ（Bedrock/Vertex 不可）
- 管理者が Auto モードを有効化している

上記をすべて満たしても利用できない場合:
- /doctor でインストール状態を診断
- Claude Code を最新版にアップデート
- 管理者に管理コンソールの設定を確認してもらう
```

### 問題が解決しない場合

上記のいずれにも該当しない場合、以下の手段を試してください。

1. **`/doctor` を実行する**: 環境全体の診断を行う
2. **セッションを新規作成する**: `/clear` で会話をリセットし、最初からやり直す
3. **アプリを再起動する**: デスクトップアプリまたは CLI を完全に終了して再起動
4. **ログを確認する**: `~/.claude/logs/` にあるログファイルを確認する
5. **フィードバックを送る**: `/feedback` で問題を報告する
6. **コミュニティに相談する**: Anthropic の公式フォーラムや Discord で質問する

### 予防策のチェックリスト

日常的に以下を実践することで、問題の発生を大幅に減らせます。

```markdown
- [ ] CLAUDE.md に明確な指示を記述している
- [ ] settings.json で危険な操作を deny に設定している
- [ ] Hook スクリプトにエラーハンドリングを含めている
- [ ] 定期的に /compact でコンテキストを圧縮している
- [ ] セッションを適切な単位で分割している
- [ ] Claude Code を最新版に保っている
- [ ] Git でこまめにコミットし、巻き戻せるようにしている
```

## やってみよう（ハンズオン）

### ステップ 1: `/doctor` を実行する

Claude Code で `/doctor` を実行し、現在の環境に問題がないか確認してください。

### ステップ 2: 予防策チェックリストを確認する

上記のチェックリストに沿って、あなたの環境の設定を点検してください。

### ステップ 3: トラブルシューティングの練習

意図的に以下の問題を発生させ、解決する練習をしてください。
- Hook スクリプトの実行権限を外して、Hook が動作しないことを確認 → 権限を付与して解決
- settings.json の deny に `Bash(git *)` を追加して、Git 操作がブロックされることを確認 → 設定を修正

---

## まとめ
- トラブルシューティングは「症状の把握 → 原因の特定 → 解決策の実行 → 再発防止」の 4 ステップ
- `/doctor` コマンドは環境全体の診断を行う最初の手段
- 主要なトラブルカテゴリは接続・パフォーマンス・ファイル操作・MCP・Hooks・Auto モードの 6 つ
- 予防策チェックリストを日常的に確認し、問題の発生を未然に防ぐ
