# コンテキスト管理 — /compact と /context

## この記事で学ぶこと
- 「コンテキスト」とは何か、なぜ管理が必要なのかを理解する
- /compact コマンドで会話を圧縮する方法を覚える
- /context コマンドでコンテキスト使用量を把握する方法を知る

## 前提知識

## 本文

### コンテキストとは

Claude Code との会話には「コンテキストウィンドウ」と呼ばれる記憶の限界があります。コンテキストとは、Claude Code が一度に覚えていられる情報の量のことです。

これを図書館にたとえると、「机の上に一度に広げられる本の数」に似ています。机が広ければたくさんの資料を参照しながら作業できますが、机の広さには限りがあるため、参照する資料が増えすぎると古いものから片づけなくてはなりません。

### なぜコンテキスト管理が大切なのか

会話が長くなると、以下のような問題が起こることがあります。

1. **記憶が薄れる**：会話の最初のほうで伝えたルールや指示を忘れてしまう
2. **応答が遅くなる**：処理するテキストが増えるほど、回答に時間がかかる
3. **コスト（利用枠）が増える**：トークン（処理するテキストの単位）の消費が増え、利用制限に早く到達する

つまり、コンテキストを上手に管理することは、Claude Code のパフォーマンスとコスト効率の両方に影響するのです。

### /context — 使用量を確認する

`/context` コマンドは、現在のコンテキスト使用量を可視化します。

```
/context
```

と入力すると、以下のような情報が表示されます。

- 現在使用しているコンテキストの割合（例：「45% 使用中」）
- 残りの余裕がどれくらいあるか
- 何がコンテキストを多く消費しているか

定期的に `/context` を確認して、使用量が多くなりすぎていないかチェックする習慣をつけましょう。目安として、70%を超えたら圧縮を検討してください。

### /compact — 会話を圧縮する

`/compact` コマンドは、これまでの会話を要約して、コンテキストの使用量を減らします。

```
/compact
```

このコマンドを実行すると、Claude Code はこれまでの会話内容を短い要約にまとめ直します。具体的なやり取りの詳細は失われますが、重要なポイント（タスクの目的、決定事項、作業状況など）は保持されます。

たとえるなら、分厚い議事録を「要点メモ」に圧縮するようなイメージです。

### /compact のオプション

`/compact` には、圧縮の際に指示を追加できるオプションがあります。

```
/compact 売上データの分析に関する情報を重点的に残してください
```

このように、残してほしい情報を指定すると、その部分を優先的に保持した圧縮が行われます。

### コンテキスト管理のベストプラクティス

#### 1. タスクの区切りで /compact する

1つのタスクが完了したら、次のタスクに移る前に `/compact` を実行しましょう。前のタスクの詳細な経緯は要約し、新しいタスクのために空きを確保します。

```
（タスクA完了後）
/compact タスクAの結果を簡潔に残してください
（タスクBを開始）
```

#### 2. 新しいトピックなら /clear を使う

前の話題とまったく関係のない新しいタスクを始めるなら、`/compact` ではなく `/clear` で完全にリセットするほうが効率的です。

```
/clear
```

これにより、コンテキストが完全にクリアされ、まっさらな状態で新しい作業を始められます。

#### 3. 長い会話を避ける設計にする

そもそもコンテキストを消費しすぎないように、以下を心がけましょう。

- **指示を具体的に書く**：あいまいな指示でやり取りが増えると、その分コンテキストを消費します
- **CLAUDE.md にルールを書く**：毎回伝え直す必要がなくなり、会話がコンパクトになります
- **大きなタスクは分割する**：1つの会話で全部やろうとせず、タスクごとに分けましょう

#### 4. /cost でコストを確認する

コンテキストの消費はコスト（利用枠）にも直結します。`/cost` コマンドで現在のトークン使用量を確認できます。

```
/cost
```

### /clear、/compact、/rewind の使い分け

この3つのコマンドは似ているようで役割が異なります。

| コマンド | 効果 | 使いどころ |
|---------|------|----------|
| /clear | 会話を完全にリセット | まったく新しいタスクを始めるとき |
| /compact | 会話を要約して圧縮 | 同じテーマの作業を続けるが、コンテキストを節約したいとき |
| /rewind | 特定の時点まで巻き戻す | 途中のステップをやり直したいとき |

## やってみよう（ハンズオン）

### ステップ1：コンテキスト使用量を確認する

現在の会話のコンテキスト使用量を確認しましょう。

```
/context
```

表示された使用量を記録しておきましょう。

### ステップ2：会話を意図的に増やす

コンテキストの変化を体感するために、いくつかの作業を Claude Code に依頼しましょう。

```
以下の5つの質問にそれぞれ詳しく回答してください。
1. プロジェクト管理で大切な3つのポイント
2. 効果的な会議のアジェンダの作り方
3. メール文章を書くときのビジネスマナー
4. データ分析の基本的な手順
5. チームでの情報共有のベストプラクティス
```

回答を受け取ったら、もう一度コンテキストを確認します。

```
/context
```

ステップ1と比べて、使用量が増えていることがわかるはずです。

### ステップ3：/compact で圧縮する

会話を圧縮してみましょう。

```
/compact プロジェクト管理とデータ分析に関する要点だけ残してください
```

圧縮後、もう一度コンテキストを確認します。

```
/context
```

使用量が減っていることを確認してください。また、圧縮で残された要点を確認するために、次のように聞いてみましょう。

```
先ほど話したプロジェクト管理のポイントを教えてください。
```

要点が正しく残っていれば、圧縮は成功です。

## まとめ
- **コンテキスト** は Claude Code が一度に覚えていられる情報の量で、限界がある
- **/context** で現在の使用量を確認できる（70%を超えたら圧縮を検討）
- **/compact** で会話を要約圧縮し、コンテキストの空きを確保できる
- **/clear** は完全リセット、**/compact** は要約圧縮、**/rewind** は巻き戻しと、それぞれ役割が異なる
- CLAUDE.md にルールを書いておくことで、会話でのやり取りを減らし、コンテキストを節約できる
